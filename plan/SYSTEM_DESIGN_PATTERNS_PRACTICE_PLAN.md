# Practice roadmap: Alex Xu system design patterns as real features

Six features, each practicing a specific pattern from _System Design Interview_ by Alex Xu,
grounded in a genuine gap identified in `SYSTEM_DESIGN_PATTERNS.md`'s "worth doing" analysis.
Built one at a time — each gets its own ADR under `docs/adr/`, its own tests, and live
end-to-end verification before being marked done here, matching every other feature in this
project's history.

- [x] **Distributed rate limiter (Redis)** — replaced the in-memory Resilience4j limiter with a
      Redis-backed sliding-window-log algorithm (sorted set + Lua script for atomicity across
      gateway replicas). `plan/DATABASE.md` named this as Redis's first intended use case;
      Redis had been running unused in `docker-compose.yml` until now. Practices: sliding-window
      log vs fixed-window vs token-bucket algorithm choice, atomic Redis operations via `EVAL`.
      Verified live: with `RATE_LIMIT_CAPACITY=3`, requests 1–3 through the real gateway
      succeeded and 4–5 got 429; the Redis sorted set was inspected directly afterward and
      contained exactly 3 timestamped, uniquely-keyed members with a TTL matching the window.
      See [ADR-0012](../docs/adr/0012-redis-backed-rate-limiter.md) (supersedes
      [ADR-0009](../docs/adr/0009-in-memory-rate-limiter.md)).
- [ ] **Cache-aside for account reads** — Redis-cache `GetAccountUseCase` reads with event-driven
      invalidation on `money-deposited`/`money-withdrawn`/`transfer-completed`, so a cached
      balance is never stale. Practices: cache-aside pattern, invalidation strategy trade-offs.
- [ ] **Notification retry + channel fallback** — `SendNotificationUseCase` currently makes
      exactly one delivery attempt per channel; add backoff retry (`resilience4j-retry`) and
      fallback to a secondary channel after retries are exhausted. Practices: Alex Xu's
      notification-system chapter directly.
- [ ] **Unique ID generation for `ledger_entries`** — add a time-ordered `sequence_id` alongside
      the existing UUID primary key, generated by a hand-rolled Snowflake-style scheme, so the
      hot-insert ledger table's index stays clustered by insertion order. Practices: distributed
      ID generation, clock-skew handling, index locality.
- [ ] **Read replica for reporting-service** — an optional RDS replica in Terraform, with
      reporting-service's read-only queries routed to it (defaulting to the primary locally,
      since there's no local replica to stand up). Practices: read/write splitting, replication
      lag, CAP trade-offs for already-eventually-consistent data.
- [ ] **Metrics alerting (Alertmanager)** — real alert rules (HTTP 5xx rate, Kafka consumer lag,
      HikariCP pool exhaustion, gateway 429 spikes) wired from Prometheus into Alertmanager.
      Practices: Alex Xu's metrics monitoring and alerting chapter.
